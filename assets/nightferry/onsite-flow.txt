# Onsite Build — End-to-End Flow (Step 0 → 22)

Working reference for the build. Describes what happens, in order, plus the
on-the-wire payloads. Two-channel model: **control** (`/jobs`, light metadata)
and **data** (`/batches`, encrypted payload). All agent calls are outbound over
HTTPS; the library network never accepts inbound connections.

---

## Vocabulary

- **Control plane** — cloud service the agent polls for work and reports status to.
- **Ingestion** — cloud endpoint that receives encrypted batches, stores them, acks.
- **Worker** — cloud process that decrypts a stored batch and applies it to the read model.
- **Mapping** — cloud-only config translating the library's schema to canonical entities. The agent never receives it; the cloud bakes it into jobs.
- **Job** — a single bounded, signed query instruction the cloud sends the agent.
- **Batch** — encrypted set of changed rows the agent uploads for one entity + sequence.
- **Agent local diff state** — agent-only fingerprints (row hashes, open-checkout set) used to compute the delta. Disposable; reseeds if lost.
- **Cloud query cursor** — per-entity read position (watermark / page) in `sync_stream_state`. Cloud-owned; drives query generation.
- **Cloud apply state** — processed batch ids + last applied sequence (`processed_batches`, `sync_stream_state`). Cloud-owned; drives idempotency and ordering.
- **Audit** — durable record of events. Authoritative copy is cloud-side.

# ACT 1 — INSTALL & SETUP (one-time)

## Step 0 — Provisioning
Concourse issues Dave the agent binary and a single-use, short-lived enrollment
code scoped to one tenant.

```
code:   MAPLE-7X9K-2233
tenant: maplewood   (bound to the code, not entered by Dave)
```

## Step 1 — Operator input
Dave runs the binary and provides SQL connection params + the enrollment code.
In production this is a 4-field wizard; for the build it is CLI flags / prompts.

```
sql_host, sql_db, sql_auth (sql|windows), sql_user, sql_password, enrollment_code
```

## Step 2 — Enrollment
Agent generates a keypair locally and registers. Private key never leaves the box.
Enrollment is idempotent: a retry with the same public key returns the same cert.

**Agent → Control `POST /enroll`**
```json
{
  "enrollment_code": "MAPLE-7X9K-2233",
  "csr": "<base64 PEM certificate signing request>",
  "agent_pubkey_fingerprint": "sha256:ab12..."
}
```

**Control → Agent `200`**
```json
{
  "agent_id": "agent-123",
  "tenant_id": "maplewood",
  "client_certificate": "<base64 PEM signed cert>",
  "endpoints": {
    "control": "https://cloud.concourse/api/control",
    "ingestion": "https://cloud.concourse/api/ingestion"
  },
  "protocol_version": 1
}
```
Agent persists cert + config locally. Code marked consumed cloud-side.
All later calls use this cert over mTLS.

**Enrollment failure handling.** The agent generates and **persists its keypair to
disk before** calling `/enroll`, then retries with that *same* key. This makes every
transient failure recoverable:

- **Call never reached the cloud** (network/timeout) -> code is still unconsumed ->
  retry with backoff (5s, 15s, 60s, then steady). The flaky modem self-heals; Dave
  does nothing.
- **Cloud signed the cert but the response was lost** -> the cloud keyed enrollment
  on `(code, key_fingerprint)`, so a retry with the same key returns the **same cert**
  instead of "code already used." (Same idempotent-retry pattern as upload dedupe,
  one layer earlier.)
- **Agent crashed mid-enrollment** -> on restart, no config -> re-runs setup -> the
  persisted keypair re-sends the same CSR -> idempotent, same cert. (This is why the
  keypair is persisted *before* the call.)
- **Genuinely bad code** (expired, claimed by a *different* key, typo) -> the cloud
  returns a clear error and the agent shows Dave a specific message ("Enrollment code
  expired, get a new one"). This is the only case Dave acts on, and it surfaces during
  setup while he is present, not silently later.

**Control → Agent `4xx` (real rejection)**
```json
{ "error": "code_expired" }   // or code_invalid | code_claimed_by_other_key
```

## Step 3 — Health check
Agent connects to SQL Server with the read-only creds and runs a trivial read.

```
ok        -> status: healthy
auth fail -> status: needs_support, reason: auth_failed
no read   -> status: needs_support, reason: no_read_permission
no host   -> status: needs_support, reason: host_unreachable
```
Surfaces problems during setup, while Dave is present.

# ACT 2 — DISCOVERY & MAPPING (one-time)

## Step 4 — Schema discovery
Agent queries SQL Server system catalog. Reports what exists; interprets nothing.

Produces an inventory:
```json
{
  "tables": [
    {
      "name": "dbo.Borrowers",
      "row_estimate": 30,
      "primary_key": ["BorrowerID"],
      "columns": [
        {"name": "BorrowerID", "type": "int", "nullable": false},
        {"name": "FullName",   "type": "nvarchar", "nullable": false},
        {"name": "EmailAddr",  "type": "nvarchar", "nullable": true},
        {"name": "Status",     "type": "varchar",  "nullable": false}
      ],
      "candidate_watermarks": ["BorrowerID"],
      "rowversion_columns": []
    }
  ]
}
```

## Step 5 — Inventory upload
**Agent → Control `POST /discovery`** — body is the inventory above.
Raw observation, not a sync config.

## Step 6 — Mapping approval (cloud-only)
Concourse derives + approves a versioned, tenant-specific mapping. Stored in the
control plane. **The agent never receives the mapping.** The cloud uses it to
generate jobs.

Mapping (lives in cloud, shown for clarity):
```json
{
  "tenant_id": "maplewood",
  "mapping_version": 4,
  "entities": {
    "patrons": {
      "source_table": "dbo.Borrowers",
      "stable_key": "BorrowerID",
      "columns": ["BorrowerID", "FullName", "EmailAddr", "Status"],
      "strategy": "snapshot_hash"
    },
    "checkouts": {
      "source_table": "dbo.Circulation",
      "stable_key": "CircID",
      "columns": ["CircID","BorrowerID","ItemID","CheckoutDate","DueDate","ReturnDate","Status"],
      "special_fields": {"return_marker": "ReturnDate"},
      "strategy": "watermark_plus_open_set"
    }
  }
}
```
If a required entity has no stable key, it is paused pending resolution.

> reads it to fill in the SQL template.

# From Enroll to Active (the poll state machine)

There is a window between "enrolled" and "syncing": the agent has uploaded
discovery, but a human has not yet approved the mapping, so no sync job can exist.
This is handled without a special endpoint — **the agent starts polling immediately
after discovery and never stops; the cloud answers based on the agent's state.**

The control plane tracks a per-agent state:
```
ENROLLED    enrolled, discovery not done yet
DISCOVERED  inventory uploaded, mapping NOT approved
ACTIVE      mapping approved, sync jobs flow
PAUSED      mapping issue or manual pause
```

The **same `/jobs/poll` endpoint** returns different things by state:
```
DISCOVERED (no mapping yet):
  -> { job: null, status: "awaiting_mapping", next_poll_seconds: 30 }
     (agent shows "waiting for Concourse setup"; cloud sets a slow wait cadence)

ACTIVE:
  -> { job: {...sync job...}, next_poll_seconds: 60 }

PAUSED:
  -> { job: null, status: "paused", next_poll_seconds: 120 }
     (agent shows "needs support")
```

Sequence:
```
1. enroll      -> cloud marks ENROLLED
2. discovery   -> inventory uploaded -> cloud marks DISCOVERED
3. agent starts the poll loop immediately
4. poll -> "awaiting_mapping, retry in 30s"   (repeats, slow cadence)
5. [human approves mapping -> cloud flips agent to ACTIVE]
6. next poll -> returns the FIRST sync job (full seed)
7. normal loop
```

Two things fall out for free:
- **No "am I ready?" endpoint.** The poll loop is the readiness check. One loop from
  enroll onward; the cloud's answer drives the agent's status.
- **The wait cadence is cloud-controlled.** While awaiting mapping the cloud sends a
  high `next_poll_seconds`, so the agent idles politely instead of hammering. This
  reuses the dynamic-polling mechanism — nothing new to build.

**First ACTIVE job per entity is the full seed — paged, not one scan.** Every job is
bounded (row limit, timeout), including the seed. A full seed means *many bounded
pages until complete*, not a single unbounded table read. `sync_stream_state` starts
at a zero cursor; each page reads `key > @cursor ORDER BY key` with a `page_size`,
the agent advances the cursor by the last key seen, and the planner keeps issuing
pages until a short page signals the end. The agent diffs each page against local
state and everything is an insert. After the seed completes, the same bounded paging
continues incrementally. No special-casing — the seed is just the incremental query
starting at a zero cursor.

# ACT 3 — STEADY-STATE LOOP (every interval, e.g. 60s)

## Step 7 — Job poll (report-and-fetch)
Agent makes an authenticated outbound request that both **reports** the last job's
result + its health and **fetches** the next job. Outbound-only = no inbound access.

**Agent → Control `POST /jobs/poll`**
```json
{
  "agent_id": "agent-123",
  "tenant_id": "maplewood",
  "supported_protocol": 1,
  "last_result": {
    "job_id": "job-788",
    "status": "success",
    "row_count": 3,
    "new_watermark": 31
  },
  "health": { "sql_pressure": "ok", "spool_pending": 0 },
  "events": []
}
```
`last_result` is omitted on the first poll. `events` carries any audit items the
agent buffered in its outbox during an outage (flushed here). The cloud reads
`last_result` -> advances the cursor + writes audit, reads `health` -> sets cadence,
then returns the next job.

## Step 8 — Job issuance
Control plane generates a signed job from the mapping + the entity's current sync
state (the planner — see "SQL Generation" below). Cloud authors the SQL; agent
receives it finished and bounded. The response also carries `next_poll_seconds`,
so the cloud controls polling cadence centrally (see "Dynamic Polling" below).

**Control → Agent `200`**
```json
{
  "job": {
    "job_id": "job-789",
    "tenant_id": "maplewood",
    "agent_id": "agent-123",
    "entity": "patrons",
    "mapping_version": 4,
    "job_version": 12,
    "mode": "snapshot_hash",
    "sql": "SELECT TOP (@page_size) [BorrowerID],[FullName],[EmailAddr],[Status] FROM [dbo].[Borrowers] WHERE [BorrowerID] > @cursor ORDER BY [BorrowerID]",
    "params": { "cursor": 0, "page_size": 500 },
    "limits": { "timeout_seconds": 5, "max_rows": 500, "max_bytes": 1048576 },
    "allowlist": { "tables": ["dbo.Borrowers"], "columns": ["BorrowerID","FullName","EmailAddr","Status"] },
    "expires_at": "2026-06-17T14:35:00Z",
    "signature": "hmac-sha256:9f2c..."
  },
  "next_poll_seconds": 60,
  "control_version": 7
}
```
If there is no work, `job` is `null` (agent just sleeps `next_poll_seconds`).
A `watermark` job (e.g. Checkouts) instead carries `"params": { "watermark": 9000, "page_size": 500 }`,
where the cloud injects the watermark from its `sync_stream_state` (see "SQL Generation").

## Step 9 — Job validation (safety gate, agent-side)
Agent validates before executing. Any failure → reject + audit + continue.

Checks:
```
- signature valid (HMAC for demo)
- tenant_id and agent_id match this agent
- not expired (expires_at)
- mapping_version is one the agent will accept
- SQL is a single read-only statement (no INSERT/UPDATE/DELETE/DDL, no ';' multi-stmt)
- every referenced table/column is in allowlist
- limits present and within local maximums (max_rows, timeout, max_bytes)
- local health ok (SQL Server not already under pressure)
```
Rejection (audited locally + reported up):
```json
{ "job_id": "job-789", "status": "rejected", "reason": "table_not_in_allowlist" }
```

## Step 10 — Query execution
Agent runs the accepted job read-only with timeout, lock timeout, page size,
streamed reads. Every job is bounded; the seed is many bounded pages until complete,
not one scan. Later runs return the windowed set the predicate defines.

## Step 11 — Change detection (agent-side diff state)
Reconcile each page against local diff state. Emit only the delta.

- `snapshot_hash` (Patrons/Holds/Fines/Books): compute per-key hash over mapped
  columns, compare to stored hashes.
  - new key  -> insert     (emit page by page)
  - changed hash -> update (emit page by page)
  - **deletes are NOT inferred from a single page.** A key missing from page 3 is not
    a delete — it may be on page 4. Deletes are only considered after a **complete
    snapshot pass**: collect the full set of seen keys across all pages, then any
    stored key not seen is a candidate delete, applied through guardrails
    (mass-delete threshold + quarantine). Inserts/updates stream; deletes wait for a
    full pass.
- `watermark_plus_open_set` (Checkouts): new rows via `CircID > watermark`;
  recheck the tracked open set for `ReturnDate` null→set. (Append-heavy: hard deletes
  are rare and handled by infrequent full-key reconciliation, not per-page.)

Delta example:
```json
{
  "inserts": [ {"BorrowerID": 31, "FullName": "A. Lee", "EmailAddr": "a@x.org", "Status": "ACTIVE"} ],
  "updates": [ {"BorrowerID": 12, "FullName": "J. Doe", "EmailAddr": "j2@x.org", "Status": "ACTIVE"} ],
  "deletes": [ 27 ]
}
```

## Step 12 — Batch construction
Pack the delta with metadata + a per-stream sequence number.

```json
{
  "batch_id": "maplewood:agent-123:patrons:5",
  "tenant_id": "maplewood",
  "agent_id": "agent-123",
  "entity": "patrons",
  "sequence": 5,
  "mapping_version": 4,
  "job_version": 12,
  "row_count": 3,
  "source_watermark": 31,
  "content_hash": "sha256:7be1...",
  "payload": "<plaintext delta, about to be encrypted>"
}
```
`batch_id` is position-based (`tenant:agent:entity:sequence`). `content_hash`
is a separate integrity check.

## Step 13 — Encrypt + spool
- generate ephemeral per-batch data key in memory
- encrypt payload (AES-GCM)
- wrap data key with cloud public material
- write {encrypted batch + sequence} to local spool in ONE transaction
- discard raw data key

Crash before commit -> nothing allocated. Crash after -> durable batch to retry.

## Step 14 — Upload
**Agent → Ingestion `POST /batches`** (mTLS)
```json
{
  "batch_id": "maplewood:agent-123:patrons:5",
  "tenant_id": "maplewood",
  "agent_id": "agent-123",
  "entity": "patrons",
  "sequence": 5,
  "mapping_version": 4,
  "content_hash": "sha256:7be1...",
  "wrapped_data_key": "<base64>",
  "ciphertext": "<base64 AES-GCM blob>",
  "nonce": "<base64>"
}
```

## Step 15 — Durable receipt + ack
Ingestion authenticates, validates tenant binding / revocation / batch_id /
sequence / content_hash, writes ciphertext to object storage under
`tenant/agent/entity/sequence`, records receipt audit, then acks AFTER durable write.

**Ingestion → Agent `200`**
```json
{ "batch_id": "maplewood:agent-123:patrons:5", "status": "stored" }
```
- same batch_id + same hash -> same ack (idempotent retry)
- same batch_id + different hash -> `409 quarantined`

## Step 16 — Spool prune + state commit
On ack: prune the spool entry (after a short retention) AND commit new sync state
(advance watermark, overwrite row hashes, bump sequence). **State commits only
after ack** — never on upload — so a lost upload never skips data.

No ack (network failure 14↔15) -> spool entry stays -> re-uploaded next loop ->
cloud idempotency makes the retry harmless. No loss, no manual step.

The result is **not** a separate call — it rides the **next** `/jobs/poll` as
`last_result` (report-and-fetch). The `new_watermark` is how the cloud advances the
**cloud query cursor**: the agent reports progress, the cloud stores it in
`sync_stream_state` and uses it to generate the next job. The cloud owns the cursor;
the agent only reports progress against it.
```json
"last_result": { "job_id": "job-789", "status": "success", "row_count": 3, "new_watermark": 31 }
```

If the report itself is lost (the poll that carried it failed), it is self-correcting:
the cursor didn't advance, so the next poll re-issues the same window, the agent
re-diffs against local state, finds no delta, and reports an empty success. The agent
also keeps unsent `last_result`/events in its outbox and re-sends on the next poll,
so nothing is dropped.

# ACT 4 — CLOUD PROCESSING (asynchronous)

## Step 17 — Processing signal
Durable write enqueues a FIFO message keyed on `tenant/agent/entity`, pointing to
the stored object.
```json
{ "object_key": "maplewood/agent-123/patrons/5", "batch_id": "maplewood:agent-123:patrons:5", "sequence": 5 }
```

## Step 18 — Worker pickup
Worker consumes the message, fetches ciphertext from object storage, decrypts via
KMS (cloud holds the private key).

## Step 19 — Validation + ordering
- already in `processed_batches`? -> skip (idempotent)
- sequence == next expected for this stream? -> if not, PAUSE stream + alert
- mapping_version current? -> if not, PAUSE stream + alert

## Step 20 — Idempotent application
Apply delta to the read model: upserts for insert/update, guarded deletes
(anomalous-share delete -> quarantine). Record `processed_batches`, advance
`sync_stream_state`, write processing audit.

## Step 21 — Serving
Portal reads only from the read model, never the source DB. Eventually consistent
(minutes), ordered, isolated, never corrupt. Source DB untouched.

## Step 22 — Continuation
Loop returns to Step 7. Each entity is an independent stream with its own sequence
and pause state; one paused stream does not block others.

# Cross-cutting: Audit

**All authoritative audit lives in the cloud.** This is forced by the Dave
constraint: nobody can read logs on the library server, so a local-only record is
unreachable and worthless. Every auditable event must reach the cloud to count.

The agent does not keep a separate authoritative audit. It **reports its events up**
to the cloud, which writes the durable record:

- agent events (job received, job **rejected** + reason, query executed, batch
  uploaded, upload failed, enrollment, health transitions) ride the next
  `/jobs/poll` as `last_result` + `events` (no separate status/events endpoint)
- cloud writes these plus its own (receipt at 15, processing/pause at 19-20) to the
  authoritative store.

The agent keeps a **local outbox** (`state/store.js`) only as a buffer: if the cloud
is unreachable when an event happens, the event is held locally and **flushed on
reconnect**. This is the same pattern as the spool — durable locally only until the
cloud has it, then the cloud owns it. The local copy is disposable.

So the safety-gate demo ("agent rejected a malicious job") is proven by **querying
the cloud audit**, not by reading a file on Dave's box. If the rejection happened
offline, it lands in the cloud audit the moment the agent reconnects.

All records hold metadata only — ids, versions, counts, hashes, status, timestamps,
error codes. **No raw patron PII.** The permanent tier is append-only with
hash-linked manifests (tamper-evident).

> Pattern (said twice in this design): the cloud owns the durable truth; the agent
> holds the minimum local state to survive an outage, then hands it off. Data uses
> the spool; audit events use the outbox.

---

# Where state lives (the boundary)

Three distinct kinds of state, named clearly so they don't blur:

| State | Name | Lives | Why |
|---|---|---|---|
| Mapping (schema → entities) | — | Cloud | interpretation; central, versioned; agent never sees it |
| Job generation / SQL authoring | — | Cloud | needs the mapping |
| **Query cursor** (watermark / page position per entity) | **cloud query cursor** | Cloud (`sync_stream_state`) | the cloud authors the next query, so it owns where to read from; advanced from the agent's reported progress |
| **Apply + sequence state** (processed batch ids, last applied sequence) | **cloud apply state** | Cloud (`processed_batches`, `sync_stream_state`) | ordering + idempotency enforced where the env is controlled |
| Authoritative audit | — | Cloud | permanent, tamper-evident |
| **Row hashes + open-set** (last-seen fingerprints for diffing) | **agent local diff state** | Agent | only the agent can see the source to diff against |
| Safety validation of incoming jobs | — | Agent | protects the on-prem DB; agent must not blindly trust the cloud |

Three names, no overlap:
- **cloud query cursor** — where to read from next (drives query generation).
- **cloud apply state** — what's been applied and in what order (drives idempotency/ordering).
- **agent local diff state** — fingerprints for change detection only.

The **agent local diff state is disposable**: if lost, the agent reseeds (re-pages
the table) and cloud idempotency dedupes the re-send. The cloud cursor and apply
state are the durable truth.

# Loop pseudocode (build target)

```
if no local config:
    enroll()              # Step 2
    health_check()        # Step 3
    discover_and_upload() # Step 4-5
    # then fall straight into the poll loop below. The cloud answers
    # "awaiting_mapping" until a human approves the mapping, then "active"
    # with real jobs. See "From Enroll to Active". No separate wait step.

flush_spool()   # on startup: re-upload any un-acked batches before new work
pending_report = None   # last job's result, sent with the NEXT poll (report-and-fetch)

while True:
    # 1. FLUSH FIRST: clear any pending spool entries before taking new work.
    #    A failed upload from a prior loop is retried here, oldest-first per stream.
    flush_spool()                             # see "Failed Upload Handling"

    # 2. REPORT-AND-FETCH: the poll carries last_result + health + buffered events,
    #    and returns the next job. No separate /jobs/status or /events call.
    resp = poll_control(last_result=pending_report,
                        health=current_health(),
                        events=outbox_drain())   # Step 7-8 -> { job, next_poll_seconds }
    pending_report = None
    interval = clamp(resp.next_poll_seconds, 5, 3600)   # cloud sets cadence, agent clamps
    job = resp.job
    if not job:
        sleep(interval); continue
    if not validate(job):                     # Step 9  (safety gate)
        pending_report = result(job, "rejected"); audit("rejected"); sleep(interval); continue

    # 3. ORDERING GUARD: do not start new work for an entity that still has an
    #    un-acked batch in the spool — its next sequence must not jump ahead.
    if spool_has_pending(job.entity):
        sleep(interval); continue             # flush_spool() above will catch up next loop

    rows = run_sql(job)                        # Step 10
    delta = diff_against_state(rows, job.entity)  # Step 11
    if delta.empty:
        pending_report = result(job, "success", row_count=0); sleep(interval); continue
    batch = build_batch(delta, job)           # Step 12  (allocates the next sequence)
    encrypt_and_spool(batch)                   # Step 13  (seq + batch in ONE txn)

    ack = upload(batch)                        # Step 14-15
    if ack.stored:
        commit_state(job.entity, batch)        # Step 16  (cursor advances ONLY here)
        prune_spool(batch)
        # staged, sent on the NEXT poll as last_result -> cloud advances cursor + audits
        pending_report = result(job, "success", row_count=batch.row_count,
                                new_watermark=batch.watermark)
        audit("uploaded")
    else:
        audit("upload_failed")                 # spool stays; cursor NOT advanced; flush next loop
    sleep(interval)

# If a poll fails, pending_report is NOT cleared -> it rides the next successful poll.
# Nothing is dropped; the report is self-correcting (lost report -> cursor unchanged
# -> next poll re-issues same window -> empty delta).


# flush_spool(): per entity stream, upload pending batches in sequence order
def flush_spool():
    for entity in streams_with_pending_batches():
        for batch in spool_pending(entity, order_by="sequence"):   # oldest first
            ack = upload(batch)
            if ack.stored:
                commit_state(entity, batch); prune_spool(batch); audit("uploaded")
            else:
                audit("upload_failed"); break   # stop this stream, keep order; retry next loop
```

# Failed Upload Handling

A failed upload is not a special error path. The spool **is** the retry mechanism:
a batch is never pruned until the cloud acks durable storage, so "failed" just means
"still in the spool, try again."

**Three failure cases, one mechanism.** All collapse to "re-upload; cloud dedupes by
`batch_id`" (batch_id is position-based `tenant:agent:entity:sequence`, so a retry
produces the identical id):
1. Upload never reached the cloud (timeout/refused) -> re-upload is a fresh insert.
2. Cloud stored it but the ack was lost -> retry hits the same batch_id -> cloud
   returns the same ack ("already stored") -> agent prunes. (This is *why* batch_id
   is deterministic, not random.)
3. Agent crashed mid-upload -> `flush_spool()` on startup re-uploads it.

**The gated-on-ack ordering (the correctness core).** On a failed upload the agent:
- does NOT prune the spool entry
- does NOT advance the cursor / `sync_stream_state`
- does NOT bump to the next sequence
- DOES write `upload_failed` to the outbox, leave everything as-is, and retry.

If the cursor advanced on a failed upload, the next job would skip the un-uploaded
rows -> permanent loss. Cursor advance and spool prune are both gated on the ack.

**Per-stream blocking.** Batches are sequenced per entity, so a stuck Patrons seq 5
must not be passed by seq 6. A pending batch **blocks its own stream** (the ordering
guard skips new Patrons work) but **other streams continue** (Checkouts is
independent). `flush_spool()` processes each stream oldest-first and stops that
stream on the first failure to preserve order.

**The spool entry is self-contained**, so a retry needs no DB re-read:
```
spool/maplewood-patrons-5.batch:
  ciphertext, nonce, wrapped_data_key
  batch_id, sequence, content_hash, metadata
  status: pending | acked
```
Retry = read file, POST again. Zero production-DB impact on retry.

**Retry timing (demo):** `flush_spool()` runs at the top of every loop, so the poll
interval is the retry interval. Production could add a few quick backoff retries
before deferring to the next loop.

**Demo script (proves the guarantee live):**
```
1. agent syncing, spool empty
2. block the cloud endpoint (kill network)
3. agent runs a job, spools a batch, upload fails
   -> show: 1 pending file in spool; cursor in sync_stream_state UNCHANGED
4. restore network
5. next loop -> flush_spool() uploads -> ack -> prune; cursor advances; read model updates
6. no rows lost, no duplicates, no human intervention
```

# SQL Generation (the planner) — cloud-side

The cloud writes the SQL. The agent never writes SQL; it runs what it is handed.
Generation lives in `cloud/control/planner.js`. It is template + mapping + cursor.

**Inputs (all cloud-side):**
1. the mapping (entity → table, key, columns, strategy)
2. the **cloud query cursor** (per-entity read position, e.g. last watermark) from `sync_stream_state`
3. the strategy (which template)

**Every template is bounded (paged).** No template is an unbounded scan — even the
snapshot reads in pages keyed on the stable key, so the seed is many bounded pages.

`snapshot_hash` — small mutable tables (Patrons, Holds, Fines, Books). No reliable
watermark, so page the table by key and let the agent diff locally.
```sql
SELECT TOP (@page_size) {columns} FROM {table} WHERE {key} > @cursor ORDER BY {key}
-- never SELECT * : always the mapped columns, to avoid noisy diffs and data leak
-- pages until a short page; deletes only after a COMPLETE pass (see Step 11)
```

`watermark` — append-heavy with a monotonic key (Checkouts new rows). The cloud
injects the cursor from the cloud query cursor.
```sql
SELECT TOP (@page_size) {columns} FROM {table} WHERE {key} > @watermark ORDER BY {key}
-- params.watermark comes from sync_stream_state (0 on first run = full seed)
```

`open_set_recheck` — the single Checkouts mutation (`ReturnDate` null→set). Targets
the agent-tracked open set in bounded chunks.
```sql
SELECT {key}, {return_marker}, {status_col} FROM {table} WHERE {key} IN (@open_ids)
```

**Decision rules:**
- `WHERE` clause is decided by `strategy` (which lives in the mapping).
- `SELECT *` never happens — always the mapped column list.
- Cursor value (`@watermark`) is owned by the cloud (`sync_stream_state`), advanced
  from the agent's reported `new_watermark`. Agent reports progress; cloud decides
  the next query.

```js
// cloud/control/planner.js (shape)
function generateJob(entity, mapping, syncState) {
  const m = mapping.entities[entity];
  const cols = m.columns.map(c => `[${c}]`).join(',');
  const table = `[${m.source_table.split('.').join('].[')}]`;
  switch (m.strategy) {
    case 'snapshot_hash':
      return makeJob(entity, m, `SELECT ${cols} FROM ${table} ORDER BY [${m.stable_key}]`, {});
    case 'watermark':
      return makeJob(entity, m,
        `SELECT ${cols} FROM ${table} WHERE [${m.stable_key}] > @watermark ORDER BY [${m.stable_key}]`,
        { watermark: syncState[entity].watermark ?? 0 });
    // open_set_recheck similar
  }
}
```

# Dynamic Polling (cloud controls cadence)

The cloud controls how often the agent polls, per tenant and per entity, with no
push and no redeploy. The cadence rides the existing poll response.

- Every `/jobs/poll` response carries `next_poll_seconds`.
- The agent sleeps that long, then polls again.
- The cloud can speed up patron data, slow catalog data, back off during
  maintenance, or surge to drain a backlog after an outage.
- If the agent reports SQL Server under pressure (in the poll's `health` field), the
  cloud responds with a larger `next_poll_seconds` — central throttle.

**Safety clamp (agent-side, same distrust boundary as job validation):** the agent
does not blindly obey. It clamps to a local floor/ceiling so a bad value
(`next_poll_seconds: 0`) cannot drive a tight loop against the library DB.
```js
const interval = clamp(resp.next_poll_seconds, 5, 3600); // MIN 5s, MAX 1h
```
Cloud proposes cadence; agent enforces local safety.

# API Endpoints (full list)

Control plane (`:3001`):
```
POST /enroll        register agent, return cert + config        (one-time)
POST /discovery     receive schema inventory                    (one-time)
POST /jobs/poll     REPORT-AND-FETCH: agent reports last result + health +
                    buffered events; cloud advances cursor, writes audit, sets
                    cadence; returns { job|null, status, next_poll_seconds }
GET  /health        sanity
```
Ingestion (`:3002`):
```
POST /batches       receive encrypted batch -> S3 -> SQS -> ack
```
Worker: no HTTP. Pulls SQS, needs no inbound.

**Why no `/jobs/status` or `/events`:** in a poll-based design the poll *is* a
report-and-fetch. The agent's last job result, health, and any outbox events buffered
during an outage all ride the next `/jobs/poll` request, so the cloud advances the
cursor, writes audit, and hands back the next job in **one round-trip** instead of
three. The data channel (`/batches`) stays separate because it carries the heavy
encrypted payload.

# Request Lifecycle Through the Code

Each endpoint mapped to the files it touches and the flow steps it implements.
Read top-to-bottom to see a request travel through the codebase.

## `POST /enroll`  (Step 2)
```
agent: setup/enroll.js
   -> HTTP POST /enroll {code, csr}
cloud: control/server.js -> routes/enroll.js
   -> validate code, mint agent_id/cert, mark code consumed
   -> shared/audit.js  (audit: enrolled)
   <- {agent_id, tenant_id, cert, endpoints}
agent: config.js  (persist cert + endpoints)
```

## `POST /discovery`  (Steps 4-5)
```
agent: setup/discovery.js
   -> db/sqlServer.js  (query system catalog -> inventory)
   -> HTTP POST /discovery {inventory}
cloud: control/server.js -> routes/discovery.js
   -> store inventory; (human approves mapping, seeded for demo)
   -> shared/audit.js  (audit: discovery_received)
```

## `POST /jobs/poll`  (Steps 7-8, 16-tail)  ← the heart of the loop (report-and-fetch)
```
agent: sync/loop.js -> sync/poller.js
   -> HTTP POST /jobs/poll { agent_id, last_result?, health, events[] }
cloud: control/server.js -> routes/jobs.js
   -> readModel.js      (apply last_result: advance cursor on success)
   -> shared/audit.js   (write audit from last_result + events — authoritative)
   -> scheduler.js      (read health -> compute next_poll_seconds)
   -> mappingStore.js   (load approved mapping)
   -> readModel.js      (read cloud query cursor)
   -> planner.js        (template + mapping + cursor -> signed SQL job)
   <- { job | null, status, next_poll_seconds }
agent: sync/validator.js   (SAFETY GATE: signature/allowlist/limits/read-only)
       sync/loop.js        (clamp next_poll_seconds 5..3600; stage next last_result)
```
One round-trip does report + audit + cadence + fetch. No `/jobs/status` or `/events`.

## query execution + upload  (Steps 9-14)
```
agent: sync/executor.js     (run bounded SQL via db/sqlServer.js)
       sync/changeDetector.js (diff vs state/store.js -> delta)
       sync/batcher.js      (delta + sequence + metadata -> batch)
       transport/crypto.js  (AES-GCM encrypt, wrap key)
       transport/spool.js   (write encrypted batch + seq, ONE txn)
       transport/uploader.js (HTTP POST /batches)
```

## `POST /batches`  (Steps 14-16)
```
agent: transport/uploader.js  -> HTTP POST /batches {ciphertext, meta}
cloud: ingestion/server.js -> routes/batches.js
   -> validate tenant/sequence/content_hash (idempotent on batch_id)
   -> shared/storage.js   (S3 put under tenant/agent/entity/sequence)
   -> shared/audit.js     (audit: receipt)
   -> shared/queue.js     (SQS send: pointer to the object)
   <- {status: stored}  (only AFTER durable S3 write)
agent: transport/spool.js (prune), state/store.js (commit cursor AFTER ack)
```

## reporting results + events  (folded into the next `/jobs/poll`)
```
# No separate /jobs/status or /events endpoint. Each job's result is staged in the
# agent and sent as `last_result` on the NEXT poll; health and any outbox-buffered
# events ride the same request. The cloud applies them (cursor advance + audit)
# before returning the next job. If a poll fails, the staged result stays in the
# agent outbox and rides the next successful poll — nothing dropped.
```

## worker (no HTTP)  (Steps 17-20)
```
cloud: worker/worker.js
   -> shared/queue.js     (SQS receive -> object pointer)
   -> shared/storage.js   (S3 get ciphertext)
   -> worker/decryptor.js (AES-GCM decrypt, local key)
   -> worker/applier.js   (check processed_batches + sequence;
                           upsert/guarded-delete to readModel.js;
                           advance sync_stream_state)
   -> shared/audit.js     (audit: processed OR paused)
   -> shared/queue.js     (delete SQS message on success)
```

**One-line trace:** `poller -> jobs route -> planner -> validator -> executor ->
changeDetector -> batcher -> crypto -> spool -> uploader -> batches route ->
storage(S3) -> queue(SQS) -> worker -> decryptor -> applier -> readModel`.

# Libraries (all mainstream + well-documented)

**TypeScript on both sides**, run via **`tsx`** (no compile step — `tsx src/main.ts`
runs TS directly). Loose `tsconfig` (types catch field/shape mistakes, not lecture).
TS is the right call here because the data contracts are already designed, Zod gives
the types for free, and across ~20 files type-checking saves more time than it costs.
Pick large, popular, heavily-documented libraries so the AI tools generate them
flawlessly.

## Shared (both sides)

| Need | Library | Why this one |
|---|---|---|
| Schemas + validation | **`zod`** | The Pydantic of TS. One schema -> runtime validation AND inferred static type. Validates every boundary payload (Job, Batch, Mapping). Huge adoption + docs. |
| Run TS without a build | **`tsx`** | Run `.ts` directly, fast iteration, no compile loop. |

## Agent (library side)

| Need | Library | Why this one |
|---|---|---|
| SQL Server driver | **`mssql`** (wraps `tedious`) | The standard Node SQL Server client. Pure JS, no system ODBC driver, bundles cleanly into the single `.exe`. Huge docs + examples. |
| HTTP to cloud | **built-in `fetch`** (Node 18+) | No dependency at all. Use `undici` only if you need fancier control. |
| Crypto (AES-GCM) | **built-in `node:crypto`** | No dependency. `createCipheriv('aes-256-gcm', ...)`. Well-documented core API. |
| Local state + spool | **plain JSON files + a `spool/` folder** | Zero native deps -> bundles into the exe with no friction. (Avoid `better-sqlite3` on the agent: it's a native module and fights single-binary packaging.) |
| CLI args / prompts | **built-in `process.argv`** or **`prompts`** | `prompts` is tiny and popular if you want interactive input; argv is zero-dep. |
| Single-binary packaging | **Node SEA** (built-in, Node 20+) or **`pkg`** | Bundles runtime + code -> one `agent.exe`. SEA is the official path; `pkg` is the battle-tested community one. Do this last. |

## Cloud side

| Need | Library | Why this one |
|---|---|---|
| HTTP framework | **`express`** | The most popular, most documented Node web framework. Routing in minutes. (`fastify` is fine too; express wins on familiarity.) |
| S3 | **`@aws-sdk/client-s3`** (v3) | Official AWS SDK. `PutObjectCommand` / `GetObjectCommand`. Verbose but everywhere. |
| SQS | **`@aws-sdk/client-sqs`** (v3) | Official. `SendMessageCommand` / `ReceiveMessageCommand` / `DeleteMessageCommand`. |
| Read model + audit DB | **`better-sqlite3`** | Synchronous, simple, extremely popular. Fine on the cloud side (native module is no problem off the agent). Swap to **`pg`** if you want real Postgres/RDS. |
| Crypto (decrypt) | **built-in `node:crypto`** | Mirror of the agent's AES-GCM. |
| Process manager (run 3 services) | **`pm2`** | Standard for keeping Node processes alive on a box. Or just tmux/screen. |

## Deliberately NOT used (and what to say)

- **No KMS / `@aws-sdk/client-kms`** — local AES-GCM key for the demo. "Wrapped by KMS in production."
- **No `pyodbc` / Python** — chose Node so the SQL Server driver (`mssql`/`tedious`) is pure-JS and the single-binary story is clean.
- **No `better-sqlite3` on the agent** — native module fights `.exe` bundling; agent state is JSON files instead.
- **No API Gateway / Lambda / ECS SDKs** — plain `express` on EC2. "Containers behind an ALB in production."
- **No Terraform** — create the bucket/queue with the AWS CLI or console. Terraform steals time for no demo value.

## One thing to verify on the day (5 min, first hour)

- **`mssql` connecting to their SQL Server** — confirm version/auth (SQL auth vs Windows auth) works against the sandbox DB early. If SQL Server isn't present or auth fights you, fall back to **SQLite as the source DB** (`better-sqlite3`) and say "abstracted the DB layer; production is SQL Server via `mssql`." Keep this fallback in your pocket; don't pre-build it.

# Schemas (Zod) — typed, validated boundaries

Every payload that crosses a boundary is defined once as a Zod schema, shared by
agent and cloud (no drift). One schema gives both **runtime validation** and the
**static TS type** (`z.infer`). This is not just hygiene — it is part of the safety
story: the agent's job validator *parses* the incoming job against `Job` before any
safety check, so malformed jobs are rejected for free; ingestion parses uploads
against `Batch` at the door; a bad seeded mapping fails loudly at load, not at query
time.

```ts
// shared/schemas.ts
import { z } from "zod";

export const Strategy = z.enum(["snapshot_hash", "watermark", "open_set_recheck"]);

export const Mapping = z.object({
  tenant_id: z.string(),
  mapping_version: z.number().int(),
  entities: z.record(z.object({
    source_table: z.string(),
    stable_key: z.string(),
    columns: z.array(z.string()).min(1),
    strategy: Strategy,
    special_fields: z.record(z.string()).optional(),
  })),
});

export const Job = z.object({
  job_id: z.string(),
  tenant_id: z.string(),
  agent_id: z.string(),
  entity: z.string(),
  mapping_version: z.number().int(),
  job_version: z.number().int(),
  mode: Strategy,
  // structured fields — the source of truth the SQL must conform to:
  table: z.string(),
  columns: z.array(z.string()).min(1),
  sql: z.string(),                // must reference only `table` + `columns`, read-only
  params: z.record(z.union([z.string(), z.number()])),
  limits: z.object({
    timeout_seconds: z.number().positive(),
    max_rows: z.number().int().positive(),
    max_bytes: z.number().int().positive(),
  }),
  allowlist: z.object({
    tables: z.array(z.string()).min(1),
    columns: z.array(z.string()).min(1),
  }),
  expires_at: z.string(),         // ISO timestamp
  signature: z.string(),
});

// JobResult is embedded in the poll request as `last_result` (not its own endpoint).
export const JobResult = z.object({
  job_id: z.string(),
  status: z.enum(["success", "rejected", "failed", "timeout"]),
  row_count: z.number().int().optional(),
  new_watermark: z.number().int().optional(),
  reason: z.string().optional(),
});

export const AuditEvent = z.object({
  type: z.string(),               // enrolled | health | rejected | upload_failed | ...
  at: z.string(),                 // ISO timestamp
  detail: z.record(z.unknown()).optional(),
});

// /jobs/poll request — report-and-fetch.
export const PollRequest = z.object({
  agent_id: z.string(),
  tenant_id: z.string(),
  supported_protocol: z.number().int(),
  last_result: JobResult.optional(),               // omitted on first poll
  health: z.object({
    sql_pressure: z.enum(["ok", "high"]),
    spool_pending: z.number().int().nonnegative(),
  }).optional(),
  events: z.array(AuditEvent).default([]),          // outbox flush
});

// /jobs/poll response.
export const PollResponse = z.object({
  job: Job.nullable(),
  status: z.enum(["active", "awaiting_mapping", "paused"]),
  next_poll_seconds: z.number().int(),
  control_version: z.number().int(),
});

export const Batch = z.object({
  batch_id: z.string(),           // tenant:agent:entity:sequence
  tenant_id: z.string(),
  agent_id: z.string(),
  entity: z.string(),
  sequence: z.number().int().nonnegative(),
  mapping_version: z.number().int(),
  content_hash: z.string(),
  ciphertext: z.string(),         // base64
  wrapped_data_key: z.string(),   // base64
  nonce: z.string(),              // base64
});

// Inferred types for free:
export type Mapping = z.infer<typeof Mapping>;
export type Job = z.infer<typeof Job>;
export type Batch = z.infer<typeof Batch>;
```

**Usage at the boundaries:**
```ts
// agent safety gate — parse first, THEN enforce safety envelope
const job = Job.parse(resp.job);          // throws on malformed job
assertReadOnly(job.sql);                  // no INSERT/UPDATE/DELETE/DDL/multi-stmt
assertAllowlisted(job);                   // tables/columns in allowlist
assertLimits(job.limits);                 // within local maximums
assertSqlMatchesFields(job);              // SQL references only job.entity's table + job.columns

// cloud ingestion — reject garbage at the door
const batch = Batch.parse(req.body);

// cloud startup — fail loudly on a bad seeded mapping
const mapping = Mapping.parse(JSON.parse(readFileSync("seed/mappings/maplewood.json")));
```

**The job carries structured fields, not just a SQL string.** This is what makes the
agent a *safety gate* rather than a fragile SQL parser. The `Job` includes `entity`,
the source `table`, the `columns`, `strategy`, `params`, and `limits` — and the
agent's check is "does the SQL reference only this table and these columns, read-only,
within these limits?" not "let me fully parse arbitrary SQL." The structured fields
are the source of truth; the SQL must conform to them. For the demo, a
keyword/allowlist check is enough; production would parse to an AST. Either way the
agent validates against the structured contract, not against the raw SQL alone.

The audit rows and local sync-state records also get small schemas, but Job, Batch,
Mapping, PollRequest, PollResponse, and JobResult are the load-bearing ones.

# Deployment

Not localhost — the agent (Windows sandbox) and cloud are separate boxes.

**Run the cloud on an EC2 instance in the AWS sandbox.** Three processes:
```
EC2 (in the sandbox VPC)
├── node src/control/server.js     :3001
├── node src/ingestion/server.js   :3002
└── node src/worker/worker.js      (no port, polls SQS)
```
Keep them up with pm2 or tmux/screen.

**Networking:**
- Same VPC as the Windows box -> agent uses EC2 private IP; security group allows
  3001/3002 from the Windows box. Cleanest, no public exposure.
- Otherwise -> EC2 public IP + security group opening 3001/3002 (lock to agent IP).

**AWS resources total:** 1 EC2, 1 S3 bucket, 1 SQS FIFO queue, 1 IAM access key.
No RDS (SQLite), no KMS (local AES), no API Gateway / Lambda / ECS (plain Node).

**Fallback if EC2 fights you:** run cloud on the laptop, `ngrok http 3001/3002`,
agent points at the ngrok URLs. Demo-grade, weaker story.

**First-hour check:** is an EC2 box pre-provisioned, and is it in the same VPC as
the Windows sandbox? That decides private vs public IP and saves setup time.

**Production story (narrate, don't build):** control plane, ingestion, and worker
are separate containers behind an ALB doing mTLS; S3 landing, SQS FIFO, RDS read
model, KMS keys. Today they are three Node processes on EC2, with real S3 + SQS for
durable handoff and async decoupling, read model and keys local, so the day goes to
the data contract rather than infra plumbing.
